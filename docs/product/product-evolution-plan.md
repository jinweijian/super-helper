# super helper 产品迭代规划

## 一句话定位

`super helper` 的第一阶段不是做另一个通用 Agent 平台，而是做企业内部的技术支持首诊助手。

它连接企业代码库、项目文档、知识库、历史工单和只读 MCP 数据源，为销售、产品、客服、技术支持新人和内部员工提供有证据、可追溯、可升级的诊断结论。

## 核心产品承诺

用户只需要描述问题，`super helper` 会：

- 判断信息是否足够诊断。
- 信息不足时追问一个最高价值问题。
- 信息足够时读取当前 workspace、知识库、历史工单或只读 MCP 数据源。
- 输出事实、推断、假设和未知。
- 给出可复制的客户回复或内部升级材料。
- 在诊断日志中保留证据、工具调用和判断过程。

## 目标用户

第一优先级用户：

- 技术支持新人：需要快速理解系统、工单、客户环境和排查路径。
- 客服和实施人员：需要先判断问题类型，减少无效升级。
- 销售：需要回答客户常见技术问题，但不能乱承诺。
- 产品经理：需要判断问题是缺陷、需求、配置问题还是使用问题。
- 技术负责人：需要减少团队被重复问题打断，并保留支持过程证据。

## 核心差异化

与 OpenClaw、Hermes 等通用 Agent 工具相比，`super helper` 的竞争点不应是更全能，而应是更适合企业支持场景：

- 场景更窄：聚焦企业内部技术支持和问题首诊。
- 证据更强：每个结论都需要来源和证据。
- 权限更稳：默认只读，不做生产修改。
- 审计更清楚：case、run、evidence、tool call 都可追溯。
- 上下文更企业化：代码库、README、历史工单、客户环境和企业 MCP 都是诊断资料。
- 输出更实用：支持客户回复、内部技术分析、升级材料和产品反馈。

## 阶段 1：本地 MVP，证明能诊断

目标：让个人或技术支持在本地绑定一个项目 workspace 后，就能完成一次有效的项目诊断。

重点能力：

- 极简对话页。
- workspace 绑定。
- Claude Code 只读诊断 worker。
- Preflight Gate：信息不足先追问，不直接派发低价值任务。
- DiagnosticRequest / DiagnosticResult 结构化。
- 诊断日志抽屉。
- 复制结论。

暂不优先做：

- 多租户。
- 复杂权限后台。
- 多 Agent 市场。
- 自动修复。
- 大而全 MCP 商店。

成功标准：

- 能基于代码库回答功能位置、配置含义和排查方向。
- 能基于 README 或项目文档回答使用问题。
- 能在用户描述不清时追问一个有效问题。
- 能输出事实、推断、未知和下一步。

## 阶段 2：证据系统，建立信任壁垒

目标：让用户相信 `super helper` 不是在猜，而是在基于证据诊断。

重点能力：

- Evidence Card：每个证据包含来源、摘要、置信度和关联结论。
- Claim Review：每个结论必须关联证据。
- Unknown 管理：明确哪些信息仍然缺失。
- 结论分层：事实、推断、假设、未知。
- 用户质疑后重新诊断，并保留旧 run。
- 诊断日志支持完整审计链路。

成功标准：

- 任意结论都能追溯到证据。
- 证据不足时不会输出最终结论。
- 同一个 case 可以多轮诊断并保留上下文。

## 阶段 3：企业知识库和历史工单接入

目标：把企业散落知识变成可诊断、可复用的上下文。

重点能力：

- 接入历史工单库，例如 `ReadReadme`。
- 接入 README、FAQ、部署文档、接口文档和故障手册。
- 支持相似案例召回。
- 输出参考历史案例的诊断建议。
- 将已解决 case 沉淀为可复用知识。

产品重点：

不要只做普通 RAG，而要做案例诊断复用。

示例输出风格：

```text
当前问题和历史工单 #2317 相似，都是课程任务保存失败。
历史原因是客户环境缺少 xx 配置。
但当前 case 尚未查询客户配置，因此这只能作为推断，不能作为最终结论。
```

成功标准：

- 能召回相似历史工单。
- 能区分历史案例参考和当前事实。
- 能把已解决问题沉淀为新的知识资产。

## 阶段 4：企业只读 MCP 生态

目标：让企业内部系统通过标准 MCP 接入诊断链路。

重点能力：

- MCP 注册和权限模型。
- 每个 workspace 配置允许使用的 MCP。
- MCP 输出统一转为 Evidence Card。
- 默认只读。
- 敏感字段脱敏。
- 工具调用日志可审计。
- 支持 `ReadShield` 查询客户服务器、环境、配置和状态。
- 支持 `ReadReadme` 查询工单库、解决方案和历史记录。

建议尽早定义公司内部 MCP 规范：

- 输入参数规范。
- 输出结构规范。
- 敏感字段规范。
- evidenceId 规范。
- 错误返回规范。
- 权限等级规范。
- 可展示字段和不可展示字段规范。

成功标准：

- 不同同事开发的 MCP 能用统一方式接入。
- MCP 结果能被 Agent 当作证据审核。
- 敏感信息不会直接暴露给不该看到的用户。

## 阶段 5：角色化工作台

目标：让同一次诊断结果服务不同岗位的真实工作。

重点能力：

- 销售输出：客户能听懂的话术，避免过度承诺。
- 产品输出：判断是缺陷、需求、配置问题还是使用问题。
- 客服输出：标准回复和下一步信息收集。
- 技术支持输出：排查路径、证据和升级材料。
- 技术负责人输出：重复问题、影响范围和产品改进线索。

同一个诊断结果可以生成：

- 客户回复版。
- 内部技术版。
- 升级工单版。
- 产品反馈版。
- 复盘总结版。

成功标准：

- 用户不需要手动改写诊断结论。
- 支持人员能直接复制客户回复。
- 技术升级时信息完整度明显提升。

## 阶段 6：团队版和服务端版

目标：让 `super helper` 从个人本地工具升级为企业内部支持入口。

重点能力：

- 多用户。
- 多租户。
- workspace 权限。
- case 分配。
- 诊断队列。
- 并发限制。
- 审计日志。
- 数据留存策略。
- 私有化部署。
- SSO 或企业账号。
- 管理后台。

成功标准：

- 企业可以统一部署和管理。
- 不同团队、客户、项目之间上下文隔离。
- 诊断过程可以被追溯和复盘。

## 阶段 7：从首诊助手升级为企业超级助手

目标：在首诊场景站稳后，再扩展到更宽的企业助手能力。

可扩展方向：

- 自动生成故障复盘。
- 自动补充知识库。
- 自动生成产品缺陷报告。
- 自动生成客户答复。
- 自动推荐负责人。
- 自动构建排查 Runbook。
- 集成飞书、钉钉、企微和工单系统。
- 从只读诊断逐步扩展到受控操作，例如创建工单、生成 SQL 检查建议、发起审批。

原则：

短期不要急着做自动修复。企业场景里，可信比自动更重要。

## 优先级建议

最高优先级：

- Preflight Gate 做扎实。
- Claude Code worker 只读诊断跑通。
- Evidence Card / Claim Review 做扎实。
- 诊断日志可审计。
- 支持 workspace + README / 工单库 / MCP 证据源。

中优先级：

- 相似工单召回。
- 可复制客户回复。
- 多轮 case 诊断。
- 用户质疑后重新 run。
- 角色化输出模板。

低优先级：

- 多 Agent 市场。
- 花哨 UI。
- 自动执行修复。
- 大而全 MCP 商店。
- 长期记忆人格化助手。

## 北极星指标

建议围绕企业支持效率定义指标：

- 技术同事被销售、产品、客服打断次数下降。
- 工单首次有效回复时间下降。
- 新人独立处理工单比例提升。
- 升级给技术时的信息完整度提升。
- 重复问题自动解决率提升。
- 诊断结论被用户质疑后的修正率下降。

## 产品判断

`super helper` 最重要的用户感受不是“这个 Agent 很全能”，而是：

```text
它真的懂我们公司的系统，而且它说的每句话都有依据。
```

只要这个感受成立，销售、产品、客服、技术支持新人都会有理由使用它，技术负责人也会有理由推动团队部署它。
